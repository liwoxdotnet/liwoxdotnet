Kubernetes is the right tool for running containerised workloads at scale. It handles scheduling, scaling, self-healing, and rolling deployments automatically. For production systems with real reliability requirements, there is no better option.

It is also genuinely complex. More complex than most teams expect before they've operated it in production.

This post covers the things we've learned running Kubernetes clusters on AWS EKS — the parts that matter in practice, not just in theory.

## Resource limits are not optional

Every container running in Kubernetes needs both resource requests and limits defined. Requests tell the scheduler how much CPU and memory to reserve. Limits tell the kubelet when to kill a misbehaving container.

Without requests, the scheduler places pods on nodes without accounting for their actual resource needs. Nodes get overloaded. Pods start evicting each other. Services go down.

Without limits, a single rogue container can consume all available memory on a node and take down every other pod running on it.

```yaml
resources:
  requests:
    memory: "256Mi"
    cpu: "250m"
  limits:
    memory: "512Mi"
    cpu: "500m"
```

Define these for every container. It is not optional configuration — it is table stakes for a stable cluster.

## Health checks are how Kubernetes keeps things running

Kubernetes has two primary health checks: liveness probes and readiness probes.

**Liveness probes** tell Kubernetes when to restart a container. If your application deadlocks but the process is still running, Kubernetes won't know without a liveness probe. With one, it restarts the container automatically.

**Readiness probes** tell Kubernetes when a container is ready to receive traffic. During a deployment, new pods come up before old ones go down. The readiness probe ensures traffic only routes to pods that have fully initialised — preventing requests from hitting containers that aren't ready.

```yaml
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10

readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
```

These two probes, properly configured, are how Kubernetes achieves self-healing. Skip them and you're operating blind.

## Namespaces are not just organisation

Most teams start with everything in the `default` namespace. This works fine until it doesn't — usually when you need to apply different resource quotas, network policies, or RBAC rules to different teams or environments.

Namespaces provide isolation boundaries. Use them deliberately: one namespace per environment (dev, staging, production), one namespace per team or service domain. Apply resource quotas at the namespace level to prevent any one team from consuming all cluster resources.

## Cluster autoscaling is not the same as pod autoscaling

The Horizontal Pod Autoscaler (HPA) scales the number of pod replicas based on CPU, memory, or custom metrics. The Cluster Autoscaler scales the number of nodes in the cluster based on unschedulable pods.

Both are necessary. HPA without Cluster Autoscaler means your pods scale but run out of nodes to schedule on. Cluster Autoscaler without HPA means your nodes scale but your pods don't distribute the load effectively.

Configure both. Set sensible minimum and maximum values. Test what happens when load increases sharply — before it happens in production.

## Observability is not optional

Running Kubernetes without proper observability is operating blind at speed. At minimum, you need:

- **Metrics** — CPU, memory, pod restarts, request rates (Prometheus)
- **Logs** — aggregated, searchable, with retention policies (Loki or CloudWatch)
- **Alerts** — firing before users notice, not after (Alertmanager)

The Prometheus + Grafana + Loki stack is the standard for a reason. It gives complete visibility at a fraction of the cost of managed monitoring services.

## The honest summary

Kubernetes delivers on its promises — self-healing workloads, zero-downtime deployments, automatic scaling, and the reliability that production systems require. But it requires real operational discipline to get those benefits. The teams that struggle with Kubernetes are usually the ones who underestimate the learning curve and skip the operational fundamentals.

Done properly, a well-configured Kubernetes cluster running on EKS can maintain 99.9% uptime with minimal manual intervention. That result is achievable — but it requires the work.
