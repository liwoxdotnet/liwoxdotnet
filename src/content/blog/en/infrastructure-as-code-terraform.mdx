Before Infrastructure as Code, provisioning a server meant logging into a console, clicking through menus, and hoping you remembered every setting the next time you needed to do it again.

Terraform made that model obsolete. Not gradually — completely.

## What Terraform actually is

Terraform is a tool for defining cloud infrastructure in configuration files. Instead of clicking through AWS, Azure, or GCP consoles, you write declarative code that describes what you want. Terraform figures out how to create it.

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"

  tags = {
    Name        = "web-server"
    Environment = "production"
  }
}
```

Run `terraform apply` and this EC2 instance exists in AWS. Run it again and nothing changes — Terraform compares your desired state against the real state and only makes the difference.

## Why it matters in production

**Reproducibility.** Every environment — development, staging, production — is created from the same code. There is no "it works on staging but not production" caused by configuration drift. The environments are identical because they are created identically.

**Version control.** Infrastructure changes go through the same review process as code changes. Pull requests, diffs, approvals, rollbacks — all the tools you use for code work for infrastructure too. You can see exactly what changed and who changed it.

**Documentation by default.** A Terraform file is documentation. Anyone joining the team can read exactly what infrastructure exists, how it's configured, and why. The alternative — infrastructure that lives only in someone's memory or in scattered console screenshots — is not documentation.

**Disaster recovery.** If you need to rebuild your entire infrastructure from scratch, you run `terraform apply`. That is the entire recovery procedure. Without IaC, a disaster recovery exercise is a multi-day manual process with significant risk of getting something wrong.

## How we use it at LiwoxDotNet

Every project we deliver includes Terraform for all infrastructure resources — VPCs, subnets, security groups, EC2 instances, EKS clusters, RDS databases, IAM roles, S3 buckets. Nothing is created manually.

We structure Terraform projects with separate state files per environment and remote state stored in S3 with DynamoDB locking. This prevents concurrent apply operations from conflicting and ensures state is always available to anyone with the right permissions.

For CI/CD integration, Terraform runs as part of the pipeline — `terraform plan` on pull requests (so reviewers can see what will change), `terraform apply` automatically on merge to main.

## The one thing most teams get wrong

The most common Terraform mistake is treating it as a one-time setup tool rather than a living part of the codebase.

Infrastructure changes should go through Terraform. Every time. If someone creates a resource manually in the console, it is invisible to Terraform's state — and the next `terraform apply` might destroy it.

Discipline here is non-negotiable. The benefit of Infrastructure as Code is only real when all infrastructure is actually in code.

## Where to start

If you're managing infrastructure manually, the fastest path to IaC is not rewriting everything at once. Start with one component — an EC2 instance, an S3 bucket, a security group — and bring it into Terraform. Import the existing resource, write the config, verify the plan shows no changes. Then expand from there.

The initial investment is hours. The ongoing return is years of reliable, reproducible, documented infrastructure.
